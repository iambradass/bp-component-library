Building Design Systems That Don't Collapse Under Their Own Weight

Design systems are supposed to make teams faster. But somewhere around version 3.0, many teams find themselves maintaining a second product that's harder to change than the one it's supposed to serve. Here's what to do differently from day one.

The warning signs come early. A designer requests a "one-time exception" to the spacing scale. An engineer adds a custom component because the existing ones "don't quite fit." Six months later, the design system has 14 button variants, three separate spacing scales, and a Figma library that bears little resemblance to the codebase. The system meant to accelerate the team is now a weight they drag.

This is not a failure of tooling. It is a failure of architecture — specifically, a failure to treat the design system as a living product with users, versioning, and a product roadmap of its own.

Start with tokens, not components

The single biggest mistake teams make is building components before establishing a token system. Tokens are the semantic layer between raw design decisions and their implementations: color-intent-primary maps to #10B981, spacing-content-gap maps to 1.5rem. This layer allows the entire system to be rethemed or updated in one place.

Without tokens, changing a brand color means finding every hex value in every component. With tokens, it means changing one variable. The investment in a three-tier token architecture — global values, semantic aliases, component-specific overrides — pays for itself the first time a rebrand lands in your roadmap.

A design system without a token layer is just a collection of components that happen to look similar today. Tokens are what make it a system.

Governance is a design problem

Most design system documentation covers components exhaustively and governance not at all. How does a new component get added? Who approves a change to an existing one? What constitutes a breaking change versus an enhancement? Without clear answers, the system accumulates debt silently, each small deviation justified in isolation, catastrophic in aggregate.

The teams with the healthiest systems treat these as design questions, not process questions. The contribution workflow should be as considered as the button component. The RFC template should be as polished as the documentation site. Every part of the system that teams interact with shapes their relationship to it.

Version aggressively, break things deliberately

Semantic versioning exists for a reason. A design system that never has breaking changes is either perfect (impossible) or is accumulating backward-compatible workarounds that make the whole thing increasingly incoherent. Teams that are afraid to increment a major version end up shipping a v1.47.0 that bears no conceptual relationship to v1.0.0.

Breaking changes are not failures. They are the mechanism by which a system improves. Document them clearly, provide migration guides, give adopters a reasonable runway, and then make the change. The teams downstream will thank you for the integrity, even if they grumble about the migration work.

Measuring what matters

How do you know if your design system is working? Most teams measure adoption rates — percentage of components from the system versus custom-built — and stop there. This is a necessary but insufficient metric. A team could have 100% adoption while still shipping slowly and inconsistently if the system doesn't serve their actual use cases.

The metrics that matter more: time from design handoff to implementation for a new page type, number of visual regressions per release, time spent per quarter on "design debt" remediation. These measure the outcomes the system exists to produce, not just the inputs.
